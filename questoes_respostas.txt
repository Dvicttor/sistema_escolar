20 perguntas respondidas para homologar e preparar o Sistema Academico para uso

Professor: Felipe Douglas
Projeto: Sistema Academico

1. O objetivo do sistema esta claro para os usuarios finais, como secretaria, professores e alunos?
Resposta: Sim. O sistema tem como objetivo cadastrar e organizar alunos, professores, disciplinas, matriculas, pesquisas e relatorios academicos.

2. Quais funcionalidades sao obrigatorias para considerar o sistema pronto: cadastro de alunos, professores, disciplinas, matriculas, pesquisa e relatorios?
Resposta: As funcionalidades obrigatorias sao: cadastrar, listar e editar alunos, professores e disciplinas; realizar matriculas; pesquisar alunos por nome; remover matriculas da interface; e gerar relatorios em JSON e PDF.

3. Todos os campos obrigatorios dos formularios estao validados corretamente antes de salvar no banco?
Resposta: Parcialmente. O banco possui campos obrigatorios com NOT NULL e valores unicos, e o sistema trata erros de integridade. Mesmo assim, e importante testar os formularios para confirmar se todos os campos obrigatorios estao impedindo envio vazio.

4. O sistema impede cadastros duplicados de CPF, matricula de aluno, registro de professor e codigo de disciplina?
Resposta: Sim. O banco usa restricoes UNIQUE para CPF, matricula, registro e codigo, e a aplicacao mostra mensagens de alerta quando ja existe cadastro duplicado.

5. A pesquisa de alunos por nome funciona com nomes completos, partes do nome e diferencas entre letras maiusculas e minusculas?
Resposta: Sim. No codigo orientado a objetos, a pesquisa usa strip() e lower(), permitindo buscar por nome completo ou parte do nome sem diferenciar maiusculas e minusculas.

6. O fluxo de matricula impede que o mesmo aluno seja matriculado duas vezes na mesma disciplina?
Resposta: Sim. A tabela matriculas possui restricao UNIQUE para aluno e disciplina, e a aplicacao verifica se a matricula ja existe antes de cadastrar novamente.

7. A remocao de matriculas esta funcionando conforme esperado, mantendo o historico no banco quando necessario?
Resposta: Sim. A exclusao de matricula nao apaga o registro do banco; ela altera o campo ativo para 0 e preenche removido_em, mantendo historico.

8. O metodo de remover dados do aluno em Aluno.py esta coerente com a regra de negocio definida para o projeto?
Resposta: Parcialmente. O metodo remove o aluno das disciplinas e limpa seus dados, mas para um sistema academico real seria melhor manter historico em vez de apagar informacoes pessoais diretamente do objeto.

9. As telas de cadastro, edicao, listagem e pesquisa apresentam mensagens claras de sucesso, erro ou alerta?
Resposta: Sim. O app.py usa flash() para mostrar mensagens como cadastro realizado, item atualizado, registro nao encontrado e dado duplicado.

10. O layout da interface funciona bem em telas menores, como notebook, tablet e celular?
Resposta: Precisa ser testado visualmente. O projeto possui arquivo de estilos, mas a homologacao deve conferir o comportamento em notebook, tablet e celular.

11. Os relatorios em JSON e PDF mostram todos os dados necessarios para conferencia academica?
Resposta: Sim. Os relatorios incluem alunos, professores, disciplinas e matriculas, com dados como nome, CPF, matricula, curso, registro, area, codigo, carga horaria, professor, alunos e status da matricula.

12. O banco MySQL pode ser criado corretamente a partir do arquivo schema.sql em um ambiente novo?
Resposta: Sim. O arquivo schema.sql cria o banco sistema_academico, as tabelas, as chaves estrangeiras, as restricoes unicas e dados iniciais.

13. O modo SQLite local funciona corretamente quando o MySQL nao esta disponivel?
Resposta: Sim. O arquivo db.py tenta conectar no MySQL e, se nao conseguir, usa automaticamente o SQLite local, criando as tabelas necessarias.

14. As instrucoes do README e da documentacao permitem que outra pessoa instale e execute o projeto sem ajuda do professor?
Resposta: Sim. O README explica como criar o banco, instalar dependencias, configurar variaveis de ambiente e iniciar a aplicacao.

15. Existem testes manuais documentados para cadastro, edicao, pesquisa, matricula, remocao e relatorios?
Resposta: Ainda precisa ser complementado. As perguntas de homologacao ajudam como roteiro, mas seria melhor criar uma tabela de testes manuais com passo a passo, resultado esperado e resultado obtido.

16. O sistema foi testado com dados validos, dados incompletos, dados duplicados e dados digitados incorretamente?
Resposta: Deve ser testado na homologacao. Pelo codigo, dados duplicados sao tratados, mas dados incompletos e incorretos precisam ser conferidos nos formularios e no banco.

17. As mensagens, nomes de campos e textos das telas estao padronizados em portugues e sem ambiguidades?
Resposta: Em geral, sim. As mensagens principais estao em portugues e sao compreensiveis, mas vale revisar acentuacao e padronizacao de termos como matricula, codigo e relatorio.

18. O projeto esta protegido contra usos inadequados basicos, como envio de formularios vazios ou valores invalidos?
Resposta: Parcialmente. O banco impede campos nulos e duplicados, mas a aplicacao ainda deve validar melhor valores invalidos, como carga horaria negativa ou textos vazios apenas com espacos.

19. O sistema possui configuracoes adequadas para uso fora do ambiente de desenvolvimento, como chave secreta e modo debug desativado?
Resposta: Parcialmente. A chave secreta pode vir da variavel FLASK_SECRET_KEY, mas existe valor padrao de desenvolvimento. Alem disso, o app.py executa com debug=True, entao para producao o debug deve ser desativado.

20. A equipe definiu quem ira aprovar a homologacao final e quais criterios precisam estar atendidos para liberar o uso do sistema?
Resposta: A equipe deve definir isso formalmente. A homologacao final pode ser aprovada pelo professor ou responsavel do projeto, considerando como criterios: cadastros funcionando, duplicidades bloqueadas, matriculas corretas, relatorios gerados, README claro e testes manuais aprovados.
